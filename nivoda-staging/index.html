OPALNOVA Nivoda Staging Handoff

Non-secret integration handoff page for Nivoda API review.

Application

Product: OPALNOVA

Platform: Windows desktop app, WPF / C#

Integration area: External supplier diamond search, saved external diamonds, quote links, hold/order tracking and received-diamond conversion.

Default Staging Endpoint

GraphQL endpoint:
https://intg-customer-staging.nivodaapi.net/api/diamonds

GraphiQL:
https://intg-customer-staging.nivodaapi.net/api/diamonds-graphiql

Credential Safety

OPALNOVA does not hardcode or publish Nivoda credentials. Test credentials must be supplied directly to the local app by the account owner or through an approved secure channel.

Hosted Review URL

This page is designed to be hosted from GitHub Pages using the repository workflow Deploy Nivoda staging handoff.

The final URL is shown by GitHub after deployment. For this repository it is expected to be either https://jdkely-dev.github.io/OpalNova/ or https://jdkely-dev.github.io/OpalNova/nivoda-staging/, depending on the chosen Pages source.

Current Implemented Surface

  • Username/password authentication against the configured Nivoda GraphQL endpoint.
  • Diamond search using lab-grown/natural, shape and carat filters.
  • Mapping of supplier diamond id, certificate, shape, carat, colour, clarity, cut, lab, price, availability, video and certificate fields where present.
  • Saving search results as OPALNOVA External Diamond records separate from owned inventory.
  • Quote-option linking, local hold/order workflow tracking, received status and conversion to owned loose-stone inventory.
  • In-app staging handoff report that can authenticate, introspect the accessible schema and list query/mutation fields without exposing credentials.
  • Static GitHub Pages handoff page for non-secret external review.

Needs Nivoda Confirmation

  1. Current staging and production endpoints for the diamond API.
  2. Current search query and filter names for lab-grown/natural, shape, carat range, lab, price, availability, certificate and video fields.
  3. Whether this account can access hold, reserve, cart, order, purchase or checkout mutations.
  4. Required mutation payloads for requesting holds, confirming orders, refreshing availability/price and retrieving supplier order documents.
  5. Any token lifetime, rate limit, permission, supplier visibility or account-region constraints.
  6. Whether Nivoda requires a webhook/callback URL. OPALNOVA is currently a local desktop app and does not expose a public callback without a separately approved hosted component.

Integration Boundary

Until Nivoda confirms account-specific mutation names and payloads, OPALNOVA should keep hold/order actions as local workflow state and require confirmation in Nivoda. Once the schema is confirmed, the app can safely add API-backed hold/order actions without changing owned-stock records prematurely.
